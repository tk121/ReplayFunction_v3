WebSocket同期の概要

本システムでは、操作画面および各VDU表示画面が ReplayWsEndpoint に WebSocket 接続する。接続時に WsHub へクライアント登録を行い、その直後に現在の Replay 状態を送信する。以後、操作画面からの制御要求や ReplayEngine による再生進行のたびに、WsHub.broadcast() により同一 roomId に属する全クライアントへ最新状態を配信する。WsHub はクライアントごとの vduNo に応じてレスポンス内容を作り分けるため、操作画面には全体状態を、VDU画面には対象VDUの個別状態を配信できる。

接続時同期

WebSocket 接続時、ReplayWsEndpoint はクエリパラメータから roomId、clientType、vduNo を取得し、WsClient として WsHub に登録する。その後 ReplaySessionService.getState() で現在状態を取得し、WsHub.sendCurrentState() により接続直後の初期状態を1回送信する。これにより、途中参加したクライアントでも現在の再生状態を即時に把握できる。

操作時同期

操作画面からの再生・停止・早送り・先頭移動・末尾移動・条件反映は HTTP API で受け付け、ReplayCoordinator.handleControl() が状態更新を行う。更新後は WsHub.broadcast() を呼び出し、操作画面自身を含む接続中クライアント全体へ最新状態を即時配信する。これにより、複数の表示画面が同じタイミングで状態変化を反映できる。

再生中同期

再生中は ReplayEngine が1秒周期で tick() を実行し、対象時間範囲のイベントを event_log から取得して順次適用する。イベント適用後は currentReplayTime を更新し、WsHub.broadcast() で全クライアントへ状態を再配信する。この仕組みにより、操作画面の状態表示と各VDUの表示内容が同一サーバ状態に基づいて同期される。

図のポイント

仕様書で補足すると分かりやすい点は次です。

WebSocketは「操作コマンド送信用」ではなく「状態配信用」

操作コマンド自体はHTTP

同期の中心はサーバ側の ReplayState

WsHub が roomId と vduNo に応じて送信先・送信内容を切り替える

途中参加した画面も接続直後に現在状態を受け取れる

この説明は、ReplayControlServlet が HTTP で制御要求を受け、ReplayWsEndpoint と WsHub が状態配信を担当する実装に一致しています。
