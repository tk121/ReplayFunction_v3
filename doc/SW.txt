2. 目的・概要

ここでは「何のためのWebサーバ機能か」を簡潔に書きます。

例としては、次のような内容です。

操作画面からの再生指示を受ける

DBに保存された操作履歴をもとに再生状態を管理する

WebSocketを通じて複数の表示クライアントへ同一状態を配信する

途中参加クライアントにも現在状態を配信する

ここは細かい実装ではなく、業務的な役割を説明する章です。

今回の同期再生Webサーバ仕様なら、特に次は必須です。

必須1: 全体処理概要

「操作画面で再生ボタン押下 → HTTP → サーバ状態更新 → WebSocketで全クライアント通知」の流れ

必須2: 起動時初期化

AppInitListener と常駐インスタンスの関係

必須3: 状態管理

ReplayState / ReplayVduState の役割分担

必須4: 通信仕様

HTTP API と WebSocket メッセージ仕様

必須5: クラス責務

各クラスの担当範囲

実際の作成順

最初から文章を全部書こうとすると大変なので、次の順で作るとやりやすいです。

手順1

まず目次だけ作る

手順2

システム構成図・処理概要図を先に作る

手順3

機能一覧を書く

手順4

HTTP / WebSocket の通信仕様を書く

手順5

クラス一覧と責務を書く

手順6

各機能の詳細処理を書く

手順7

最後にエラー処理、ログ、非機能を補完する

仕様書の粒度

仕様書には粒度がいくつかありますが、今回おすすめなのは次の中間レベルです。

画面仕様書ほど利用者寄りではない

詳細設計書ほど1メソッド単位ではない

保守担当者がソースを読まなくても全体と責務がわかる粒度

つまり、

クラス単位

API単位

処理フロー単位

で書くのがちょうどよいです。
